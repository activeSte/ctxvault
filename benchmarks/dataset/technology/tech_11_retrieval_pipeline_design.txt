Title: Retrieval Pipeline Design for Production Systems

Building a production retrieval pipeline requires careful consideration of multiple stages. The first stage is document ingestion, where raw documents are parsed, cleaned, and normalized. Different file formats such as PDF, DOCX, and markdown each require specialized extractors that preserve structural information while removing formatting artifacts.

The second stage is chunking, which determines how documents are split into retrievable units. Naive fixed-size chunking often breaks semantic boundaries, splitting sentences or paragraphs in the middle. More sophisticated approaches use recursive splitting that respects paragraph and sentence boundaries, or structure-aware splitting that follows document headings and sections. The choice of chunk size involves a fundamental tradeoff: smaller chunks improve retrieval precision because each chunk is more topically focused, but they lose context that may be needed for the language model to generate accurate responses. Larger chunks preserve more context but may dilute the relevance signal during vector similarity search.

The third stage is embedding, where each chunk is converted into a dense vector representation. The embedding model must be chosen carefully. General-purpose models like sentence-transformers work well for broad domains, but domain-specific fine-tuned models can significantly improve retrieval quality for specialized corpora. Embedding dimensionality also matters: higher dimensions capture more nuance but increase storage and computation costs.

The fourth stage is indexing, where embeddings are stored in a vector database for efficient similarity search. Approximate nearest neighbor algorithms like HNSW provide sub-linear query time at the cost of slight recall reduction. The index must be configured with appropriate parameters for the expected corpus size and query latency requirements.

The fifth stage is query processing. When a user query arrives, it goes through the same embedding model used for document chunks. The resulting query vector is compared against stored vectors using cosine similarity or inner product distance. Advanced systems add query expansion, where the original query is augmented with related terms, or hypothetical document embeddings (HyDE), where an LLM first generates a hypothetical answer that is then used as the search query.

The final stage is post-retrieval processing. Retrieved chunks are re-ranked using a cross-encoder model that considers the full query-document interaction, not just independent embeddings. The top chunks are then assembled into a context window for the language model, with careful attention to token budgets and information ordering. Some systems add a citation step where each claim in the generated response is linked back to specific source chunks for verifiability.
